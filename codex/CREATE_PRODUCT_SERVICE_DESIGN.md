# Passvero CreateProduct Application Service Design

## 1. Purpose and Use-Case Boundary

`CreateProduct` creates one usable Product aggregate for the active
Organization. One successful invocation creates, atomically:

1. one stable Product;
2. one initial editable ProductVersion;
3. one source-locale ProductTranslation;
4. the Product `currentDraftVersion` pointer; and
5. one append-only AuditLog event.

The service is the only owner of this business operation. It does not create an
empty Product for a later request to repair, expose generic Product creation,
or depend on HTTP, UI, authentication-library, or Prisma details.

It does not create Passport, QRCode, ProductIdentifier, ProductMaterial,
ProductDocument, ProductImage, Document, Notification, IntegrationMapping,
BackgroundJob, Subscription, or storage/provider records.

## 3. Trusted Context and Permission

The service requires an `AuthenticatedUserContext` constructed by trusted
server infrastructure. It supplies:

- authenticated `userId`;
- active `organizationId`;
- the eligible ACTIVE Membership identity;
- resolved Membership role and permissions; and
- `correlationId`.

The exact required permission is `PRODUCT_CREATE`. The canonical matrix grants
it to EDITOR, ADMIN, and OWNER, but the service authorizes the permission rather
than hard-coding role comparisons.

The context is not command data. Client-supplied Organization, User, actor,
Membership, or role values never override it.

## 4. Command Contract

The conceptual command is intentionally narrow:

| Field | Requirement | Meaning |
| --- | --- | --- |
| `initialLocale` | Required | Source authoring locale for the initial draft and its translation. |
| `initialProductName` | Required | Initial localized product name and source for the stable internal label at creation. |
| `organizationSku` | Optional | Organization-managed SKU; blank input is treated as absent. |

The presentation boundary rejects unknown fields and applies bounded string
limits before invoking the service. The service repeats security-relevant
normalization and rejects empty required values.

`initialLocale` is trimmed and must be a supported BCP 47 or approved Passvero
locale code. `initialProductName` is trimmed, must remain non-empty, and is
stored as the source-locale `productName`.

The initial Product `internalName` is derived from the same normalized
`initialProductName`. This avoids asking the first workflow step for two names
while still populating the required stable operational field.

Initial localized descriptions, identifiers, and materials are not command
fields. Their dedicated draft-content services own them later.

## 5. Derived and Forbidden Inputs

### Classification

| Potential value | Classification | Decision |
| --- | --- | --- |
| Initial locale | Required | Caller selects the source authoring locale. |
| Initial product name | Required | Creates one meaningful source translation. |
| Organization SKU | Optional | Normalized by the service; may be absent. |
| `internalName` | Derived by service | Uses normalized initial product name at creation. |
| Stable public code | Derived by service | Generated by a trusted opaque-code generator. |
| Product lifecycle status | Derived by service | `ACTIVE`. |
| ProductVersion status | Derived by service | `DRAFT`. |
| Version number | Derived by later publication | `null` during draft creation. |
| Initial descriptions | Deferred | Owned by later draft-content editing. |
| Initial identifiers | Deferred | Owned by ProductIdentifier draft services. |
| Initial materials | Deferred | Owned by ProductMaterial draft services. |

The caller must not provide:

- any database ID;
- `organizationId`, Product owner, ProductVersion owner, or actor ID;
- `publicCode`;
- lifecycle or version status;
- version number or label;
- current draft or published pointer;
- `createdById`, `updatedById`, `publishedById`, or `archivedById`;
- clone source;
- publication, review, supersession, discard, archive, or other timestamps;
- Passport or QRCode values; or
- audit fields or metadata.

## 6. Initial Product State

The final Product state is:

| Product field | Initial value |
| --- | --- |
| `organizationId` | Trusted context Organization. |
| `internalName` | Normalized `initialProductName`. |
| `sku` | Trimmed caller SKU or `null`. |
| `normalizedSku` | Service-derived canonical SKU value or `null`. |
| `publicCode` | Newly generated opaque stable code. |
| `lifecycleStatus` | Exact enum value `ACTIVE`. |
| `currentDraftVersionId` | New initial ProductVersion ID after guarded assignment. |
| `currentPublishedVersionId` | `null`. |
| `createdById` | Authenticated `userId`. |
| `updatedById` | Authenticated `userId`. |
| `archivedById`, `archivedAt` | `null`. |
| `lastPublishedAt` | `null`. |
| `createdAt`, `updatedAt` | Server/database-controlled creation time. |

The Product is briefly inserted with both pointers null only inside the owned
transaction. That intermediate state is never committed or returned.

The stable Product identity, Organization ownership, and `publicCode` do not
change through ordinary draft or publication workflows.

## 7. Initial ProductVersion State

The initial ProductVersion:

- belongs to the newly created Product;
- derives `organizationId` from trusted context and keeps it equal to
  `Product.organizationId`;
- uses exact status `DRAFT`;
- uses normalized `initialLocale` as `sourceLocale`;
- has `versionNumber = null` because publication assigns the next
  product-scoped number transactionally;
- has `versionLabel = null` and `changeSummary = null`;
- has `clonedFromVersionId = null`;
- derives `createdById` and initial `updatedById` from authenticated `userId`;
- has `publishedById = null`;
- has `reviewReadyAt`, `publishedAt`, `supersededAt`, and `discardedAt` null;
  and
- uses server/database-controlled creation and update timestamps.

It is the only DRAFT or READY_FOR_REVIEW ProductVersion for the Product. The
partial unique index remains the final database guard.

## 8. Initial Versioned Content Decision

`CreateProduct` creates exactly one ProductTranslation for the initial
ProductVersion:

- `locale` equals normalized `ProductVersion.sourceLocale`;
- `productName` equals normalized `initialProductName`;
- every optional translated content field is null; and
- timestamps are server/database controlled.

This is the narrowest meaningful initial draft supported by the current
workflow: the source language and product name exist immediately, while the
draft may remain incomplete.

Descriptions, identifiers, materials, documents, and images are deferred to
purpose-specific draft-content services. They are not convenience work inside
this transaction.

## 9. Authorization and Validation Flow

The service performs these stages in order:

1. require a valid `AuthenticatedUserContext`;
2. validate and normalize the command;
3. require an ACTIVE Membership eligible for the active Organization;
4. require `PRODUCT_CREATE`;
5. require the Organization to be `ACTIVE` and eligible for normal mutation;
6. derive Product, ProductVersion, actor, and audit values;
7. begin one transaction;
8. revalidate Organization, Membership, and permission-sensitive facts that
   may have changed;
9. validate SKU availability as a preliminary clarity check when present;
10. execute the complete create-then-point sequence; and
11. map the committed result.

Validation remains separated:

- input validation handles shape, supported locale, bounds, and nonblank name;
- authorization handles identity, ACTIVE Membership, tenant, and permission;
- business validation handles Organization eligibility and aggregate state;
- persistence constraints arbitrate races and structural integrity.

No Product-count or subscription entitlement check is added. Current
authorities do not define an approved CreateProduct limit policy. If one is
later approved, server-side entitlement evaluation belongs before mutation and
must be revalidated as required.

## 10. Transaction Boundary and Sequence

`CreateProduct` owns one business transaction:

1. revalidate the trusted Organization as ACTIVE and the Membership as ACTIVE
   with `PRODUCT_CREATE`;
2. validate normalized identity values and, when supplied, inspect
   Organization SKU availability;
3. generate and validate a candidate stable public code immediately before the
   Product insert;
4. create Product with both current-version pointers null;
5. create the initial ProductVersion under the same Product and Organization;
6. create the required source-locale ProductTranslation;
7. assign `Product.currentDraftVersionId` only when it is still null and the
   version belongs to that Product and Organization;
8. append the Product-created AuditLog event; and
9. read the narrow fields required for the result.

Every operation uses the same explicit transaction-scoped persistence
capability. A failure or recognized collision rolls back Product,
ProductVersion, ProductTranslation, pointer assignment, and AuditLog together.
A confirmed public-code uniqueness violation aborts that PostgreSQL transaction;
the service never continues work in the failed transaction. A permitted
collision retry starts the complete sequence again in a new transaction with a
fresh candidate and revalidates concurrency-sensitive authorization and
Organization eligibility.

The transaction performs no storage, network, provider, queue, email,
notification-delivery, or detached background work.

## 11. Persistence Operations

Future persistence infrastructure may provide purpose-specific capabilities
equivalent to:

- verify the Organization and Membership facts required for Product creation;
- find a Product by trusted Organization and normalized SKU;
- create the stable Product identity with derived ownership and provenance;
- create its initial ProductVersion;
- create its source ProductTranslation;
- assign the current draft only when unset and ownership matches;
- append the Product-created AuditLog event; and
- read the created aggregate projection required by the result.

These are conceptual responsibilities, not method names or interfaces.

Persistence operations preserve the service-owned transaction and report
narrow known outcomes. They do not authorize, choose lifecycle values,
generate public codes, decide idempotency, coordinate the workflow, expose a
root Prisma client, or offer generic create, update, save, upsert, or
unrestricted mutation operations.

## 12. Stable Product Code Generation

`Product.publicCode` stores the stable public Product code. It is a required
PostgreSQL `TEXT` value protected by the implemented global unique index. The
schema has no conflicting length or format constraint.

The approved format is exactly:

- 16 cryptographically secure random bytes;
- 128 bits of entropy;
- RFC 4648 base64url encoding without `=` padding;
- exactly 22 ASCII characters; and
- alphabet `A-Z`, `a-z`, `0-9`, hyphen (`-`), and underscore (`_`).

The code contains no Product or Organization ID, timestamp, sequence number,
SKU, slug, locale, checksum suffix, client prefix, human-readable Product data,
or other tenant or business information.

Generation is server-only through a dedicated generator. The command cannot
supply, prefix, or override the code. The generated value is validated against
the approved 22-character format before persistence.

The code is opaque and difficult to enumerate, but it is not an authentication
secret. Possession of it never authorizes a private operation. It may appear in
the intended public route and QR target, while logs and telemetry avoid
recording it unless an explicitly reviewed operational need requires doing so.

The value is immutable after Product creation. It remains stable across draft
creation and editing, publication, archival, restoration, Passport changes,
and QR operations. Ordinary Product updates never regenerate it.

The database's global `Product.publicCode` uniqueness is authoritative. Only a
positively identified conflict on that exact uniqueness guarantee permits a
new candidate. Repeated collisions indicate a generator, deterministic
test-double, or constraint-mapping defect rather than expected random behavior.

## 13. Organization SKU Rules

`organizationSku` is optional.

For the MVP contract:

- trim leading and trailing whitespace;
- convert an empty or whitespace-only value to `null`;
- store the trimmed display value in `Product.sku`;
- derive `Product.normalizedSku` through one shared SKU normalizer;
- preserve case for the current canonical value; do not universally lowercase
  or perform undocumented Unicode transformations; and
- enforce uniqueness within `organizationId`.

Thus the current database comparison remains case-sensitive. Any future
case-insensitive policy is a behavior and data-migration decision, not a silent
normalizer change.

A preliminary scoped lookup may provide a clear error, but the
`(organizationId, normalizedSku)` unique constraint decides concurrent races.
A known duplicate maps to `CONFLICT`, not `VALIDATION`. Multiple null
normalized SKU values are allowed.

## 14. Concurrency and Idempotency

### Duplicate SKU

Two concurrent commands with the same Organization and normalized non-null SKU
may both pass a preliminary check. Only one may commit. The known Product SKU
constraint maps the loser to a safe duplicate-SKU `CONFLICT`.

### Public-code collision

One service invocation permits exactly three total public-code candidates:

1. generate a candidate immediately before Product insertion;
2. if insertion fails with the positively identified Product public-code
   uniqueness conflict, abandon the failed transaction;
3. start the complete business transaction again and generate a fresh,
   independent candidate;
4. repeat once more after a second confirmed collision; and
5. after the third confirmed collision, stop and return a safe `INTERNAL`
   application error.

No candidate is reused after a confirmed collision. Every failed attempt leaves
no Product, ProductVersion, ProductTranslation, pointer, or AuditLog row.

Public-code retry does not apply to duplicate Organization SKU, validation or
authorization failure, unrelated transaction conflict, another uniqueness
error, unknown database failure, or network failure. Classification must match
the exact Product public-code uniqueness guarantee; broad "unique error"
handling is prohibited.

Public-code collision classification and generic transient transaction
classification are separate policies. They do not nest or reset one another.
The CreateProduct coordinator permits at most three total transaction
executions and therefore at most three generated candidates for one service
invocation. A permitted generic transient retry consumes the same overall
execution budget rather than creating an inner retry loop. A public-code
collision consumes one candidate and is never also classified as a generic
transient retry.

### Duplicate command retry

CreateProduct has no service-level idempotency key for the MVP. The frozen
schema contains no command-idempotency record, and an optional SKU cannot serve
as command identity.

Consequences are explicit:

- a duplicate command with the same non-null SKU conflicts;
- a retry without SKU may create another Product;
- presentation adapters should prevent accidental double submission but
  cannot claim semantic idempotency; and
- durable retry-after-unknown-commit semantics require a separately reviewed
  idempotency design before being promised.

### Draft pointer assignment

The pointer update requires `currentDraftVersionId` to remain null and verifies
the new version's Product and Organization ownership. A zero-row or unexpected
existing pointer is an internal `INVALID_STATE`; the complete transaction
rolls back.

General transient database retries are bounded and allowed only when the whole
transaction is safe to rerun. No external effect occurs inside it.

## 15. Audit Event

The transaction appends one AuditLog record:

| Audit field | Value |
| --- | --- |
| `organizationId` | Trusted active Organization. |
| `actorId` | Authenticated `userId`. |
| `action` | Canonical `PRODUCT_CREATED`. |
| `entityType` | Canonical `PRODUCT`. |
| `entityId` | New Product ID. |
| `summary` | Short non-localized safe statement that a Product was created. |
| `correlationId` | Trusted context correlation ID. |
| `occurredAt`, `createdAt` | Trusted/server-controlled time. |

Metadata is either absent or restricted to an allowlist such as the initial
ProductVersion ID and a boolean indicating whether an SKU was supplied. It
must not contain the command payload, product descriptions, personal data,
headers, session data, secrets, storage data, or provider data.

Audit insertion is mandatory and atomic. AuditLog remains append-only.

## 16. Result Contract

The service returns a minimal command result, not a broad draft summary:

| Result field | Meaning |
| --- | --- |
| `productId` | New Product ID. |
| `initialProductVersionId` | New initial draft ID. |
| `publicCode` | Stable Product public code. |
| `productStatus` | Exact value `ACTIVE`. |
| `draftStatus` | Exact value `DRAFT`. |
| `organizationSku` | Stored trimmed SKU or `null`. |
| `createdAt` | Product creation timestamp. |

This is sufficient for the next adapter to navigate to or load the draft
through a separate authorized query. It excludes raw Prisma records,
Organization internals, actor data, Membership data, translation content,
audit metadata, and persistence implementation details.

## 17. Error Contract

Expected application outcomes are:

| Category | CreateProduct meaning |
| --- | --- |
| `UNAUTHENTICATED` | No valid authenticated trusted context. |
| `FORBIDDEN` | Context lacks eligible ACTIVE Membership or `PRODUCT_CREATE`. |
| `NOT_FOUND` | Previously resolved Organization context is stale or no longer safely resolvable. |
| `VALIDATION` | Invalid command shape, blank name, unsupported locale, excessive length, or invalid SKU syntax. |
| `CONFLICT` | Known duplicate Organization SKU or future incompatible idempotency identity. |
| `INVALID_STATE` | Organization is ineligible or guarded pointer/aggregate state is unexpected. |
| `LIMIT_EXCEEDED` | Reserved for a future approved server-side Product limit; not emitted by the current design. |
| `INTERNAL` | Three confirmed public-code collisions, unavailable database, or unrecognized persistence failure. |

Persistence translation recognizes the exact approved Product SKU,
public-code, active-draft, pointer, foreign-key, and CHECK outcomes. It does not
convert every unique or database error into `CONFLICT`.

Raw Prisma errors, codes, SQL, constraint names, connection details, stack
traces, and cross-tenant identifiers never escape.

## 18. Security Requirements

- Accept only the three allowlisted command fields.
- Derive tenant, ownership, actor, lifecycle, pointers, and timestamps.
- Load and mutate with trusted Organization scope.
- Revalidate ACTIVE Membership and `PRODUCT_CREATE`.
- Never authorize through caller-provided IDs, roles, or ownership.
- Never return Prisma Product, ProductVersion, ProductTranslation, or AuditLog
  records.
- Treat the public code as a public opaque locator, never as authorization for
  private reads or mutations.
- Generate the public code only on the server and never accept a caller
  override.
- Keep the code immutable after creation and avoid unnecessary logging.
- Keep AuditLog metadata minimal and allowlisted.
- Log only safe categories, timing, counters, and correlation ID.
- Expose no private content through a public DTO.
- Provide no hidden administrative, import, worker, or maintenance bypass.
- Require platform tools and imports to use this service or an explicitly
  reviewed invariant-equivalent boundary.

## 19. Dependencies and Ports

Conceptual dependencies are:

- an application-owned transaction runner/unit-of-work boundary;
- purpose-specific Product aggregate persistence capabilities;
- trusted Organization and Membership fact loading;
- deterministic permission policy for `PRODUCT_CREATE`;
- stable public-code generator backed by 16 cryptographically secure random
  bytes and unpadded RFC 4648 base64url encoding;
- defensive public-code format validation;
- exact public-code uniqueness-conflict classification;
- bounded public-code-attempt coordination;
- locale and SKU normalization policies;
- trusted clock;
- append-only AuditLog persistence; and
- protected observability.

There is no idempotency port in the MVP design.

Dependencies exclude HTTP requests, Next.js request objects, cookies, forms,
authentication SDKs, provider SDKs, storage, notification delivery, queues,
and a globally exposed Prisma client. Simple deterministic normalization
helpers do not require speculative interfaces.

## 20. Testing Contract

### Unit tests

Unit tests cover:

- command allowlisting and required-field validation;
- locale, name, and SKU trimming;
- blank SKU to null;
- case-preserving SKU canonicalization;
- `PRODUCT_CREATE` policy use;
- public-code output length of exactly 22 characters;
- exclusive use of the approved base64url alphabet;
- absence of base64 padding;
- independent values across repeated generation;
- absence of embedded Product or Organization data;
- rejection of malformed generator output before persistence when the
  defensive validation boundary is used;
- exactly three total public-code generation attempts;
- derived Product and ProductVersion values;
- minimal result mapping; and
- exact known persistence-error translation.

### Service tests

Service tests prove authorization precedes mutation, one transaction owns all
writes, every dependency receives the same persistence scope, pointer
assignment is guarded, audit is mandatory, no external dependency is called,
and raw persistence records/errors do not escape. Deterministic generator test
doubles drive collision paths without weakening production cryptography.

### PostgreSQL/Prisma integration tests

Release-gate integration tests prove:

- successful Product, DRAFT ProductVersion, source ProductTranslation, pointer,
  and AuditLog creation;
- Product, ProductVersion, and AuditLog use the trusted Organization;
- Product and ProductVersion creators use authenticated `userId`;
- final `currentDraftVersionId` points to the new version;
- `currentPublishedVersionId`, publication time, and archive state remain null;
- exact initial statuses, null `versionNumber`, and timestamps are correct;
- translation locale equals `sourceLocale` and its name is normalized;
- no identifiers, materials, Passport, QRCode, Document, ProductDocument,
  ProductImage, Notification, IntegrationMapping, BackgroundJob, Subscription,
  or storage record is created;
- failure during ProductVersion, translation, pointer, or audit insertion
  leaves no Product aggregate rows;
- unauthorized, SUSPENDED, and REMOVED Memberships are rejected;
- missing permission is rejected;
- stale or ineligible Organization is rejected;
- caller-supplied tenant, actor, lifecycle, and pointer fields cannot change
  derived values;
- duplicate and concurrent duplicate SKU attempts return one safe conflict;
- null SKUs may coexist;
- generated public code is stored unchanged, returned unchanged, and cannot be
  supplied or overridden by the caller;
- the first confirmed public-code conflict starts a new transaction with a new
  code;
- the second confirmed conflict starts a third transaction with a third code;
- the third confirmed conflict returns a safe `INTERNAL` failure;
- duplicate Organization SKU does not invoke public-code retry;
- unrelated uniqueness failures do not invoke public-code retry;
- unknown Prisma failures do not invoke public-code retry;
- every failed insertion transaction leaves no partial aggregate or AuditLog;
- a successful later candidate creates exactly one Product aggregate;
- public-code and transaction retry counters remain independently bounded;
- retry telemetry contains no generated code, Prisma code, constraint name, or
  SQL detail;
- active-draft and pointer constraints cannot be bypassed;
- repository/application boundaries expose no generic write bypass; and
- result fields are exactly allowlisted.

### Idempotency and concurrency tests

Tests explicitly prove the MVP non-idempotent behavior: a duplicate non-null
SKU conflicts, while commands without SKU are not represented as retries of
one logical command. No test may describe UI duplicate prevention as service
idempotency.

All applicable Product ownership, cross-tenant rejection, AuditLog append-only,
transaction, and service-invariant groups are release gates.

## 21. Observability

Safe signals are:

- success count;
- failure count by stable application category;
- service and transaction duration;
- transient transaction retry count;
- public-code collision count;
- public-code collision-exhaustion count;
- duplicate-SKU conflict count; and
- correlation ID.

Telemetry does not include command payloads, product names, SKUs, descriptions,
User data, session data, raw identifiers beyond approved protected correlation,
candidate public codes, Prisma errors or codes, constraint names, SQL, or
secrets. Collision exhaustion records a safe categorized failure and
correlation ID without database internals. No telemetry provider is selected
here.

## 22. Proposed Module Location

The future use case should live conceptually under:

`src/application/products/create-product/`

That module may separate command, service/use-case, result mapping, focused
policies, and tests where repository conventions justify it. Exact filenames
are deferred until the implementation plan inspects established source
patterns.

Prisma persistence belongs under the approved server-only infrastructure
boundary, not inside the application module.

## 23. Definition of Done

CreateProduct is complete only when:

- command and minimal result contracts match this design;
- trusted context and exact `PRODUCT_CREATE` permission are enforced;
- Organization and Membership eligibility are revalidated;
- Product, ProductVersion, source ProductTranslation, pointer, and AuditLog are
  created in one transaction;
- all tenant, actor, lifecycle, and timestamp values are derived;
- exact `ACTIVE` and `DRAFT` states and null publication state are preserved;
- version number remains null until publication;
- pointer assignment is guarded;
- public code is generated from 16 cryptographically secure random bytes and
  stored as exactly 22 unpadded base64url characters;
- public code is server-generated, opaque, immutable, and never used as private
  authorization;
- the exact Product public-code uniqueness conflict is translated narrowly;
- no more than three independent public-code candidates are generated;
- every collision retry begins a new complete transaction;
- public-code and generic transaction retry policies remain separately and
  jointly bounded;
- third-collision exhaustion returns safe `INTERNAL` and secret-free telemetry;
- MVP non-idempotency is not misrepresented;
- no unrelated aggregate or external effect is created;
- raw Prisma errors and records cannot escape;
- no generic repository bypass exists;
- all required unit, service, PostgreSQL/Prisma, concurrency, rollback,
  authorization, security, and allowlist tests pass; and
- documentation remains synchronized.
