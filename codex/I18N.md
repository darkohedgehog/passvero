# Internationalization

## Library

Use `next-intl`.

## Supported locales

```ts
['hr', 'sr', 'en', 'de', 'sl', 'pl']
```

Languages:

- `hr` — Croatian
- `sr` — Serbian, Latin script
- `en` — English
- `de` — German
- `sl` — Slovenian
- `pl` — Polish

## Default locale

Croatian:

```ts
defaultLocale: 'hr'
```

## URL strategy

Use:

```ts
localePrefix: 'as-needed'
```

Expected behavior:

```txt
/       Croatian
/en     English
/de     German
/sr     Serbian
/sl     Slovenian
/pl     Polish
```

Croatian should normally be served without `/hr`.

## Required i18n files

```txt
messages/
├── hr.json
├── sr.json
├── en.json
├── de.json
├── sl.json
└── pl.json

src/i18n/
├── routing.ts
├── request.ts
└── navigation.ts
```

Use the proxy or middleware convention supported by the installed Next.js version. Do not create both.

## Message rules

- All locale files must share the same key structure.
- Croatian is the canonical source language unless the task says otherwise.
- Serbian translations must use Latin script.
- Do not place long page copy directly inside components.
- Use namespaces by domain.

Recommended namespaces:

```json
{
  "Metadata": {},
  "Navigation": {},
  "Common": {},
  "Home": {},
  "Auth": {},
  "Dashboard": {},
  "Products": {},
  "Passports": {},
  "Validation": {},
  "Errors": {}
}
```

## Translation quality

Translations must sound natural.

Avoid literal word-for-word translation when it produces unnatural phrasing.

Use correct terminology for:

- Digital Product Passport;
- manufacturer;
- importer;
- distributor;
- materials;
- repairability;
- spare parts;
- recycling;
- compliance;
- product origin.

## Navigation

Use helpers from `src/i18n/navigation.ts` for locale-aware navigation.

Do not manually manipulate locale prefixes with string replacement.

A locale switcher must:

- preserve the current route;
- preserve dynamic route parameters;
- remain keyboard accessible;
- display the current language;
- support all six locales.

## Locale validation

Validate route locales against the shared routing configuration.

Unsupported locales must return the application 404 behavior.

Do not dynamically import arbitrary paths from user input.

## Static rendering

Use `setRequestLocale` when required by the installed `next-intl` version.

Avoid request-only APIs when static rendering is possible.

## Metadata

Use localized metadata.

Do not invent canonical production URLs. Only configure canonical and alternate links when a reliable application base URL exists.

## Dates, numbers and currency

Use locale-aware formatting.

For prices in the EU, use EUR where relevant.

Never hardcode English date formatting.

## Fallback behavior

Croatian is the explicit fallback locale.

Do not silently mix languages on a page.
